## 2024-04-12 - Making custom div elements keyboard accessible
**Learning:** When using non-native HTML elements like `<div>` as interactive components (e.g. project cards acting as accordions), they lack native keyboard accessibility. Specifically, screen readers don't announce them as interactive, they aren't part of the document's tab order, and they don't respond to standard keyboard events (Enter/Space) to trigger their actions. Simply relying on click listeners breaks keyboard navigation for a large segment of users.

**Action:** Always ensure that custom interactive components emulate native controls by applying the proper ARIA roles (e.g. `role="button"`), making them focusable with `tabindex="0"`, providing state information with `aria-expanded` (if applicable), and explicitly listening for `keydown` events checking for 'Enter' and 'Space' keys to trigger the same logic as the `click` event. Additionally, it is critical to implement a robust `*:focus-visible` global CSS rule so users can visually track their focus ring across the page.
