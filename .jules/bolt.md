## 2024-04-19 - Optimizing Math.sqrt in RequestAnimationFrame Loops
**Learning:** In performance-critical `requestAnimationFrame` loops, doing mathematical square roots (`Math.sqrt`) globally for variables like velocity, or locally for multiple elements' distances, is highly inefficient. We found that calculating distances inside an interaction threshold wastes CPU cycles if the object isn't even close to the mouse.
**Action:** Pre-calculate the squared threshold constants outside the loop (e.g., `radiusSq = radius * radius`). Within the loop, compute the squared distance (`distSq = dx*dx + dy*dy`) first. Use a simple `distSq < radiusSq` check to determine if the element is within range. If the actual distance is needed for continuous gradients/lerping, only execute `Math.sqrt(distSq)` *inside* this true condition block. Furthermore, wrap global variables like velocity that require `Math.sqrt` in lazy-loading closures (e.g., `getVelocity()`) so they are only calculated if at least one element actually needs them.
